## 1. 游戏概述

> **本章导读：** 这一章讲清楚「我们在做什么游戏」—— 一句话定义、核心里循环、以及 16 个周末后的成功标准。读完你应该能判断这个项目适不适合你。

### 一句话定义

**正俯视 2D 暗黑 Like 刷宝游戏**—— 砍怪 → 爆装 → 变强 → 砍更强的怪。灵感来自《暗黑破坏神》系列，但简化到只剩最核心的「刷宝快感」。

### 核心里循环

```
进入地牢 → 砍怪 → 怪物掉落装备 → 捡起装备看属性 → 换上更强的 → 砍更难的怪
                 ↑                                                      │
                 └────────────────── 变强了，回来继续 ──────────────────┘
```

这个循环是游戏**唯一的核心**。每砍死一只怪，你都有可能得到一件更好的装备；每换上一件装备，你都能直观地感觉到自己变强了。这就是 Diablo-like 让人上瘾的根本原因。

### 游戏体验定位

| 维度 | 定位 | 说明 |
|------|------|------|
| 玩法 | 砍杀 + 刷宝 | 没有解谜、没有平台跳跃、没有叙事选择 |
| 难度曲线 | 渐进 | 前期 1-2 刀一只怪，后期需要策略走位 + 技能 |
| 单局时长 | 15-30 分钟 | 下地牢 → 刷 → 回城卖垃圾 → 再来 |
| 美术风格 | 像素风混搭 | 免费素材拼贴，不追求统一画风 |
| 目标玩家 | 你自己 | 先做出自己愿意玩的东西 |

### 成功标准

16 个周末后，有一个可玩的 demo：

- 单一地牢（固定地图，不是随机生成）
- 3 种以上怪物类型
- 5 级品质装备系统 + 背包 + 装备栏
- 商店 NPC（卖垃圾换金币）
- 基础技能（火球术）
- 升级系统（经验条 + 属性成长）
- Boss 战
- **总游戏时长：能连续玩 30 分钟不无聊**

## 2. 关键设计决策

> **本章导读：** 每个游戏项目在起步时都有很多「方向选择」。这一章记录了为什么选 A 而不是选 B，避免以后 回头后悔。这些决策不是随便做的——每一个都有明确的成本收益考量。

| 决策 | 选择 | 理由 |
|------|------|------|
| **视角** | 正俯视 (Top-Down) | 开发难度最低，素材匹配最容易，碰撞检测内置。不需要处理跳跃重力这些复杂物理 |
| **美术** | 免费素材包混搭 | 零美术基础。Kenney.nl + OpenGameArt 足够覆盖角色、怪物、TileMap、图标 |
| **装备品质** | 绿→蓝→紫→金→暗金 | WoW 系经典配色，5 级玩家一眼能分辨，视觉层次足够 |
| **地图生成** | 单一固定地牢 | v1 先跑通所有系统。随机地牢（AutoTile / 程序生成）留给 v2 |
| **学习方式** | 模块化跟写 + 独立拼合 | 每个系统独立学教程 → 理解透 → 拼到一起。比「跟着一个完整项目抄」理解更深 |
| **核心里程碑优先级** | 爆装快感 > 战斗深度 > 画面打磨 | 这是游戏最上瘾的部分。宁可背包界面丑一点，也要先做出「掉落→捡起→装备→变强」的循环 |
| **输入映射** | 自定义 `right/left/down/up` | 不使用 Godot 默认的 `ui_*` 前缀映射，避免跟 UI 导航冲突 |
| **Player 节点类型** | CharacterBody2D | 内置 `move_and_slide()` + 碰撞检测，比 Area2D 更简单，比 RigidBody2D 更可控 |

### 3.2 玩家状态机

#### 状态定义

玩家使用枚举定义 4 个状态：

```gdscript
enum State {
    IDLE,    # 待机：不移动、不攻击
    RUN,     # 移动：WASD 按住时
    ATTACK,  # 攻击：鼠标左键按下后，持续到攻击动画播完
    DEAD     # 死亡：HP ≤ 0（待实现）
}
```

#### 状态转换图

```
State.IDLE  ─── WASD 按下 ──────→ State.RUN
State.RUN   ─── WASD 全松 ──────→ State.IDLE
State.IDLE  ─── 鼠标左键 ───────→ State.ATTACK
State.RUN   ─── 鼠标左键 ───────→ State.ATTACK
State.ATTACK ── timer(0.6s) 超时 → State.IDLE
State.DEAD  ─── (HP ≤ 0) ──────→ (待实现)
```

#### 状态转换条件 & 保护逻辑

| 当前状态 | 触发条件 | 目标状态 | 保护逻辑 |
|----------|---------|---------|---------|
| IDLE | 按 WASD 任意键，`move_direction != Vector2.ZERO` | RUN | 只在 IDLE 时转换，防止 RUN→RUN 重复触发 |
| RUN | 所有 WASD 松开，`move_direction == Vector2.ZERO` | IDLE | 同上 |
| IDLE / RUN | 鼠标左键按下 (`_unhandled_input`) | ATTACK | **关键保护：** `attack()` 第一行检查 `if state == State.ATTACK: return`，防止攻击动画未播完时再次攻击（连点鼠标不会打断当前攻击） |
| ATTACK | `await create_timer(attack_speed).timeout` | IDLE | 攻击结束后**强制回到 IDLE**，不能从 ATTACK 直接回 RUN（角色攻击后需要收招，用户体验上更清晰） |
| DEAD | 待实现 | — | 死亡状态下所有输入无效 |

#### 代码中的保护机制

**`_physics_process` 层的状态守卫：**

```gdscript
func _physics_process(_delta: float) -> void:
    if not state == State.ATTACK:
        movement_loop()
```

ATTACK 状态下**不允许移动**。角色站在原地完成攻击动作后才能移动。这是很多动作游戏的基本设计——攻击有「硬直」。

**`attack()` 函数中的重复攻击保护：**

```gdscript
func attack() -> void:
    if state == State.ATTACK:
        return  # 攻击动画正在播放，忽略本次输入
```

如果玩家狂点鼠标，只有**第一次**触发攻击。后面 N-1 次点击都被忽略。

#### ⚠️ 已知 bug：状态机第 41 行逻辑错误

```gdscript
# ❌ 错误写法（当前代码）
if state == State.IDLE or State.RUN:

# ✅ 正确写法
if state == State.IDLE or state == State.RUN:
```

**影响：** `State.RUN` 是一个非零枚举值（值为 1），在 GDScript 中被视为 `true`。所以这个条件永远成立。当前 flip_h 在任何状态下都会执行，但在 ATTACK 时 `move_direction` 仍保留上次的移动方向，所以视觉上没有明显 bug。但语义上不对，应在攻击状态时使用攻击方向来计算翻转。

### 3.3 动画系统

#### 整体架构

```
AnimationTree (root: AnimationNodeStateMachine)
│
├── 状态节点:
│   ├── idle    ← 待机动画（SpriteSheet 帧 0-5）
│   ├── run     ← 跑步动画（SpriteSheet 帧 6-11）
│   └── attack  ← 这是一个 BlendTree，内部包含 BlendSpace2D
│
├── 状态转换:
│   ├── Start → idle (自动进入)
│   ├── idle ↔ run (依赖 state 变化，通过 travel())
│   ├── idle → attack (鼠标左键)
│   ├── run → attack (鼠标左键)
│   └── attack → idle (timer 超时自动回)
│
└── 控制方式:
    animation_playback.travel("状态名")
```

`AnimationNodeStateMachinePlayback.travel()` 是 Godot 提供的状态机播放控制方法。调用它会自动处理入场/出场动画混合，不需要手动管理动画切换的插值逻辑。

#### BlendSpace2D 详解

**为什么需要 BlendSpace2D？**

角色可以向 8 个方向攻击，但 SpriteSheet 只存储了 4 个方向的攻击动画帧。BlendSpace2D 的作用是：**根据攻击方向（2D 向量）自动选择对应的动画**。

**工作原理：**

```
┌─────────────────────────────────┐
│         BlendSpace2D            │
│   (输入: blend_position, 2D 向量) │
│                                 │
│    (-1, 0) ← attack_left        │
│         ↑                       │
│    (0, -1) ← attack_up          │
│         ↓                       │
│    (0, 1)  ← attack_down        │
│         →                       │
│    (1, 0)  ← attack_right       │
│                                 │
│    (中间值自动插值混合!)           │
└─────────────────────────────────┘
```

**方向到帧的映射：**

| 方向 | blend_position | SpriteSheet 帧区域 | 动画剪辑名 |
|------|---------------|-------------------|-----------|
| 下攻击 (down) | (0, 1) | 行 4, 帧 24-29 | `attack_down` |
| 左攻击 (left) | (-1, 0) | 行 2, 帧 12-17 | `attack_left` |
| 右攻击 (right) | (1, 0) | 行 2, 帧 12-17 (flip_h) | 复用 `attack_left` |
| 上攻击 (up) | (0, -1) | 行 6, 帧 36-41 | `attack_up` |

**为什么右攻击复用左攻击的帧 + flip_h？**

Warrior_Blue.png 的 SpriteSheet 只绘制了向左的侧面攻击动画。通过水平翻转 (`Sprite2D.flip_h = true`)，向左的动画帧可以镜像为向右的动画，节省 SpriteSheet 空间。这在 2D 像素游戏里是非常常见的做法。

**攻击方向计算：**

```gdscript
var mouse_pos: Vector2 = get_global_mouse_position()
var attack_dir: Vector2 = (mouse_pos - global_position).normalized()
animation_tree.set("parameters/attack/BlendSpace2D/blend_position", attack_dir)
```

每次攻击时，计算从玩家位置到鼠标位置的向量，归一化后写入 BlendSpace2D 的 blend_position。Godot 自动选择最近的攻击方向动画。

#### 朝向翻转 (flip_h) 逻辑

**移动时的翻转：**

```gdscript
if state == State.IDLE or state == State.RUN:
    if move_direction.x < -0.01:
        $Sprite2D.flip_h = true   # 朝左
    elif move_direction.x > 0.01:
        $Sprite2D.flip_h = false  # 朝右
```

使用 `-0.01` / `0.01` 阈值而不是 `== 0`，是因为浮点数精度问题。避免玩家轻微摇杆抖动时频繁翻转。

**攻击时的翻转：**

```gdscript
$Sprite2D.flip_h = attack_dir.x < 0 and abs(attack_dir.x) >= abs(attack_dir.y)
```

这个条件做了两件事：
1. **判断水平朝向：** `attack_dir.x < 0` — 鼠标在玩家左侧就朝左
2. **防止斜向时翻转错误：** `abs(attack_dir.x) >= abs(attack_dir.y)` — 只在水平分量大于等于垂直分量时才翻转。如果鼠标在左上角（x=-0.6, y=-0.8），垂直分量更大，此时不翻转（由上攻击动画覆盖）

## 4. 装备系统设计

> **本章导读：** 装备系统是这个游戏「让人上瘾」的核心。这一章不只说技术设计——还要说玩家在游戏过程中**会感受到什么**。从「打怪掉绿装」到「全身暗金」，每一阶段的体验都应该不同。

### 4.1 品质等级

装备品质是这个游戏最直接的「反馈信号」。每掉落一件装备，玩家第一眼看的就是**颜色**。

| 品质 | 颜色 | 属性倍率 | 典型掉落率 | 玩家看到时的心理反应 |
|------|------|---------|-----------|-------------------|
| **绿色** (Uncommon) | `#2ecc71` 🟢 | 1.0× | 50% | 「哦，又一件绿装，看看属性……还行，分解货」 |
| **蓝色** (Rare) | `#3498db` 🔵 | 1.5× | 30% | 「不错，比我身上的好一点，换上」 |
| **紫色** (Epic) | `#9b59b6` 🟣 | 2.0× | 12% | 「卧槽紫装！这刀攻击力直接翻倍」 |
| **金色** (Legendary) | `#f1c40f` 🟡 | 3.0× | 6% | 「发财了！！金装！！截图发朋友！」 |
| **暗金** (Unique) | `#e67e22` 🟠 | 5.0× | 2% | 「……这是真的吗？暗金？？？」 |

**掉落率是叠加计算的：** 每击杀一只怪，掉落一件装备的概率系统会按上表比例抽取。绿色 50% → 蓝色 30% → 紫色 12% → 金色 6% → 暗金 2%。平均每 50 只怪出一件暗金。

### 4.2 玩家升级体感变化

装备的「变强感」是关键。玩家在不同阶段应该有这样的体验：

#### 🟢 前期（1-10 级 / 绿蓝装阶段）
- **感觉：** 砍怪从 3 刀变 2 刀，自己挨打从 5 刀死变 8 刀死
- **爆装频率：** 每 3-5 只怪掉一件，且大部分是绿装
- **反馈设计：**
  - 捡到蓝装时，装备图标有微光特效
  - 装备后攻击力数字跳动上升（有 +5 的飘字动画）
  - 伤害数字从白色变为淡黄色（表示你变强了）

#### 🟣 中期（11-20 级 / 紫装起步）
- **感觉：** 开始有「build」的概念了——堆攻击力还是堆暴击？
- **爆装频率：** 绿装开始变少，蓝紫装比例上升
- **反馈设计：**
  - 紫装掉落时屏幕边缘有紫色闪光
  - 装备上的属性多了一条随机词缀（如 "+3% 暴击"）
  - 玩家可以明显感觉到自己比 10 级时强了 3-5 倍

#### 🟡🟠 后期（21-30 级 / 追金装暗金）
- **感觉：** 每一件装备的提升都很珍贵，属性开始「乘法叠加」
- **爆装频率：** 金装仍然稀缺，暗金是终极追求
- **反馈设计：**
  - 金装掉落时全屏闪烁 + 音效（待实现）
  - 暗金有独特的装备外观（变色/粒子特效）
  - 穿 3 件以上同一套装有额外效果（套装系统 v2 做）

### 4.3 物品类型

**v1（当前里程碑 3 范围）：**

| 类型 | 具体物品 | 主要属性 | 掉落位置 |
|------|---------|---------|---------|
| **武器** | 剑、斧、弓 | 攻击力 (ATK) | 所有怪物 |
| **护甲** | 头盔、胸甲、鞋子 | 防御力 (DEF) | 精英怪掉落率更高 |
| **饰品** | 戒指、项链 | 血量加成 / 暴击率 | Boss 掉落 |
| **消耗品** | 血瓶、蓝瓶 | 瞬间恢复 HP/MP | 宝箱、商店 |

**v2 扩展：**
- 前缀/后缀随机词缀系统（如「锋利的火焰之剑」= +攻 +火焰伤害）
- 套装效果（2 件套 / 4 件套额外属性）
- 装备等级要求（防止低等级穿高装）

### 4.4 掉落流程

```
怪物死亡
    ↓
查 DropTable → 决定掉什么物品类型（武器? 护甲? 消耗品?）
    ↓
随机抽取品质（按概率表，50%绿 / 30%蓝 / 12%紫 / 6%金 / 2%暗金）
    ↓
品质倍率 × 基础属性 = 最终属性（绿1.0× → 暗金5.0×）
    ↓
生成 Item 节点在地面上
    ↓  (待实现)
发光效果（品质越高光效越亮）
    ↓  (待实现)
玩家靠近 + 按 E → 捡起进背包
```

### 4.5 核心类设计

| 类名 | 类型 | 职责 |
|------|------|------|
| `Player.gd` | CharacterBody2D 脚本 | 移动、攻击、HP、属性计算、装备属性加成 |
| `Enemy.gd` | CharacterBody2D 脚本 | AI 移动、攻击判定、HP、掉落逻辑 |
| `ItemData.gd` | **Resource** | 物品模板（名称、基础属性、品质、图标路径） |
| `Inventory.gd` | Node / Singleton | 背包数组管理、装备/卸下、属性重算 |
| `DropTable.gd` | Resource | 配置怪物掉落池：什么怪掉什么类型、概率多少 |
| `DamageNumber.gd` | Node2D | 伤害数字飘字动画 |

## 7. 避坑指南

> **本章导读：** 这些坑都是开发过程中**真实踩过的**。遇到类似问题时，先来这里查一下——很可能已经有人替你踩过了。

| 坑 | 症状 | 原因 & 解决 |
|---|------|------------|
| **没有主场景** | 按 F5 报错 `no main scene` | Project → Project Settings → Application → Run → Main Scene → 选择 `test_map.tscn` |
| **角色穿墙** | 走到地图边缘直接穿出去 | 确保 Player 是 CharacterBody2D，墙是 StaticBody2D。**两种物理体类型匹配才能碰撞** |
| **角色不动** | 按 WASD 没反应 | 确认 player.tscn 的**根节点**附加了 player.gd 脚本。检查 `_physics_process` 中是否在调用 `movement_loop()` |
| **信号不触发** | `connect()` 写的回调函数没有被调用 | 用 Godot 编辑器 Node → Signals 面板连接，不要手写 `connect()` 字符串路径。字符串路径在重构节点名时会断 |
| **@export 改了但无效** | 属性面板改了值，运行游戏还是旧值 | 改完 @export 变量后 **Ctrl+S 保存场景**。@export 变量的值存储在 .tscn 文件中不保存就不会更新 |
| **UI 不显示** | 背包/血条节点在场景中但看不见 | UI 节点必须在 **CanvasLayer** 下，而不是 Node2D 下。CanvasLayer 是独立的渲染层，不受游戏世界坐标影响 |
| **WASD 没反应（输入映射）** | 按方向键角色不动 | 当前使用自定义 `right/left/down/up` 映射（不是默认的 `ui_right/ui_left`）。打开 Project → Input Map 检查是否已添加 |
| **Camera2D 不跟随** | 角色走出屏幕 | 选中 Camera2D 节点，在 Inspector 中勾选 **Current** 属性。没有勾选 Current 的 Camera2D 不会被视口使用 |
| **SpriteSheet 显示错误** | 角色显示成一整张图或者只显示第一帧 | 在 Sprite2D 属性中设置 **hframes**（水平帧数）和 **vframes**（垂直帧数）。Warrior_Blue 是 6×8 |
| **Image width 0 报错** | 输出面板出现 `Image width is 0` | `assets/environment/` 下有损坏的图片文件。在 FileSystem 面板中逐个选中右键 → Reimport 找到坏文件 |
| **405 个 TileMap 报错** | 打开 test_map.tscn 时大量报错 | 残留了超出图片范围的无效 Tile 定义。手动删除了 1279 行无效 `.sub_resource` 定义修复 |
| **NVIDIA 不生效（Wayland）** | 游戏运行卡顿，用的还是核显 | `prime-run godot --editor` 启动编辑器，或用 `envycontrol -s nvidia` 切到独显模式。CachyOS 下 Wayland + NVIDIA 是已知坑 |

## 8. 每周开发节奏建议

> **本章导读：** 你不是全职做这个游戏——你是周末才有时间。这一章给出一个「可持续的」周末开发流程，让你每次坐下都能有效推进，而不是花半小时想「我上次做到哪了」。

### 标准周末安排

```
周六（3-4 小时）：
  ├── 1. 看教程（30-60 分钟）
  │      不要一次看太多。看一个完整的、跟本周任务匹配的教程
  │      收藏起来，下次继续看
  ├── 2. 跟着做（1-2 小时）
  │      边看边写，不要跳步。遇到看不懂的暂停思考
  └── 3. 自己改（30 分钟）
        把教程的代码改成自己的版本
        换参数、换变量名、试试「如果这里改成 X 会怎样」

周日（2-3 小时）：
  ├── 1. 回顾上周代码（15 分钟）
  │      打开上周写的代码 —— 还能看懂吗？
  │      看不懂说明需要加注释
  ├── 2. 继续做（1-2 小时）
  │      完成本周剩下的任务
  └── 3. 运行测试（15 分钟）
        F5 运行一遍，确认能跑通
        不能跑通 → 修复后再结束
```

### 核心规则

> **每次只学一个系统。** 不要同时改「移动 + 战斗 + 掉落」。做完一个，运行一次。确认跑通，再做下一个。

这就是模块化开发 —— 每个模块独立可验证。如果一次改了三个系统然后出 bug，你根本不知道是哪个改坏了。

### 周日收尾检查清单

每周末结束时，过一遍这个清单：

- [ ] F5 运行游戏，能正常启动
- [ ] 本周新增的功能能工作（哪怕很丑）
- [ ] 上周的功能没有被破坏
- [ ] 有 bug 的地方记下来了（写备忘录或 Issues）
- [ ] 代码提交了（`git add -A && git commit -m "wip: 本周做了什么"`）
- [ ] 下次要做什么有个大致计划

## 9. 架构原则

> **本章导读：** 这些都是踩坑换来的经验，写给未来的你。当你犹豫「代码应该放哪里」的时候，回来看看这一章。

### 原则 1：每个场景做一件事

```
Player       → 移动 + HP + 攻击
Enemy        → AI 行走 + 攻击 + 掉落
Inventory    → 背包管理 + 装备/卸下
DropTable    → 怪物掉落概率计算
```

**不要**把所有代码写在一个文件里。Godot 的场景系统就是为了让你拆分的。

### 原则 2：用信号通信

```
✅ Player 发出 "attacked" 信号 → Enemy 监听并减血
❌ Player 直接调用 enemy.take_damage()
```

这是 Godot 的最佳实践。好处是：
- Player 不需要知道它砍到的是什么（敌人、箱子、NPC？）
- 增加新类型的可交互对象时不需要改 Player 代码
- 便于调试——信号可以被单独连接和测试

### 原则 3：物品用 Resource 定义

```
❌ 在代码里硬编码：{name: "铁剑", atk: 10, quality: "blue"}
✅ 创建 ItemData Resource，在编辑器里填属性，存为 .tres 文件
```

**为什么？**
- Resource 文件可复用（一个铁剑模板 → N 把铁剑掉落实例）
- 编辑器里可视化编辑（拖拽设置图标、填数值）
- 修改属性不需要改代码，改 .tres 文件即可

### 原则 4：先跑通，再优化

> **第一版代码丑没关系。** 跑通了再重构。

新手最容易犯的错误：在「能跑」之前就开始纠结「这样写对不对」。不要在一个系统还跑不通的时候就考虑性能优化。先让角色能走路，再考虑走路姿势好不好看。

### 原则 5：不做超前设计

```
❌ "未来可能会需要随机地牢，所以现在先预留接口……"
✅ 今天只做今天需要的功能
```

16 周的时间很有限。每次打开编辑器之前，先问自己：「做完这件事，游戏更好玩了吗？」如果答案是否定的——它属于「到时候再说」的范畴。
